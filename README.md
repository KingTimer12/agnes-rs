# Agnes

**A type-safe database toolkit for TypeScript with a Rust core and a built-in, self-invalidating query cache.**

Agnes is what you get when you take the ergonomics of a modern TypeScript ORM
(schema-as-code, typed queries, typed relations) and put a **Rust engine**
underneath it — connection pooling, SQL parsing, and a content-addressed cache
all run natively through a [napi-rs](https://napi.rs) bridge, not in JavaScript.

```ts
import { db } from "./db";              // generated by `agnes generate`
import { gt } from "agnes-library";

const adults = await db
  .select("user")
  .where(gt(usersCol.age, 18))
  .include({ posts: true })   // typed nested relation, no N+1
  .ttl(60)                    // cached for 60s, auto-invalidated on write
  .all();
```

---

## What makes Agnes different

Most TypeScript ORMs (Prisma, Drizzle, TypeORM) run the whole query lifecycle in
JavaScript and leave caching to you. Agnes moves the hot path into Rust and makes
caching a first-class, **correct-by-default** feature.

### 1. Rust core, TypeScript surface

The query executor, connection pool (`sqlx`), SQL parser (`sqlparser`), and cache
live in Rust crates and are exposed to Node/Bun via a single native addon. You
write TypeScript; the heavy lifting never touches the JS event loop.

```
your app (TS)  →  agnes-library (typed DSL)  →  agnes-bridge (napi)  →  Rust core
                                                                          ├─ executor
                                                                          ├─ parser (sqlparser)
                                                                          ├─ cache (WAL)
                                                                          └─ adapter (sqlx): pg / mysql / sqlite
```

### 2. A query cache that invalidates itself — correctly

This is the headline feature. Other tools give you a cache and make *you*
remember to bust it. Agnes does it automatically and safely:

- **Automatic on reads.** Any `SELECT` is cached transparently. Set a TTL per
  query with `.ttl(secs)`.
- **Content-addressed keys.** Keys are a [BLAKE3](https://github.com/BLAKE3-team/BLAKE3)
  hash of the **normalized** SQL + bound params — so `SELECT * FROM users` and
  `select  *  from  users` hit the same entry, and different params never collide.
- **Tag-based invalidation.** Every query is parsed to extract the tables it
  touches (`tbl:users`, …). On any `INSERT`/`UPDATE`/`DELETE`, Agnes invalidates
  exactly the tags for the affected tables — no manual cache keys, no stale reads.
- **WAL-backed & durable.** The cache is a write-ahead log (`.agnes/cache.wal`)
  with compaction, so it survives restarts instead of being a cold in-memory map.
- **Opt-out per query** with `.bypassCache()` when you need the source of truth.

You get Redis-style read caching with none of the "did I forget to invalidate
that?" footguns — because invalidation is derived from the SQL itself.

### 3. Type-safe schema **and** typed relations

Define your schema once as TypeScript. Column types, nullability, and relations
all flow into inference — including nested `.include()` results:

```ts
export const schema = {
  user: table({
    id:    int("id").primary(),
    email: text("email").uniqueIndex("email_idx"),
    active:bool("active").default(true),
    posts: many("post", "userId"),
  }, "users"),
  post: table({
    id:      int("id").primary(),
    userId:  int("user_id"),
    user:    one("user", "userId", "id", OnAction.None, OnAction.Cascade),
  }, "posts"),
};
```

`db.select("user").include({ posts: true })` returns
`{ ...user, posts: Post[] }` — fully typed, resolved with a single batched `IN`
query per relation (no N+1).

### 4. One API, three databases

PostgreSQL, MySQL, and SQLite behind the same query builder. Placeholders,
identifier quoting, and DDL dialect differences are handled for you.

### 5. A real schema CLI

`agnes-cli` covers the full schema lifecycle — no hand-written migrations
required:

| Command | What it does |
|---------|--------------|
| `agnes push` | Sync the DB to match `schema.ts` (create/alter/**drop**, full mirror; confirms destructive ops) |
| `agnes pull` | Introspect a live DB and regenerate `schema.ts` |
| `agnes migrate` | Diff schema vs DB → write a versioned `.sql` → apply pending, tracked in `_agnes_migrations` |
| `agnes generate` | Emit a ready-to-import client module (`db.ts`/`db.js`) wired from your config |

FK-aware ordering, dialect type mapping, and destructive-op confirmation are
built in. See [`agnes-cli/README.md`](./agnes-cli/README.md).

## Quick start

```bash
bun install
bun run build       # build the native addon (agnes-bridge)
bun run build:lib   # build agnes-library
```

Create `agnes.config.ts`:

```ts
import { defineConfig } from "agnes-cli";
import { schema } from "./src/schema";

export default defineConfig({
  driver: "sqlite",
  url: "sqlite:./demo.db",
  schema,
  output: "./src/db.ts",                 // where `agnes generate` writes the client
  schemaPath: "./src/schema.ts",
  cache: { enabled: true, walPath: ".agnes/cache.wal" },
});
```

Then:

```bash
bun run agnes push        # create tables from schema.ts
bun run agnes generate    # write src/db.ts (pre-wired AgnesClient)
```

```ts
import { db } from "./src/db";
await db.insertInto("post").values({ userId: 1, content: "Hello!" });
```

## Packages

| Package | Language | Role |
|---------|----------|------|
| `agnes-core` | Rust | Executor, SQL parser, cache trait, cache-key hashing, value types |
| `agnes-cache` | Rust | WAL-backed cache backend with tag invalidation + compaction |
| `agnes-adapter-postgres` / `-mysql` / `-sqlite` | Rust | `sqlx`-based dialect adapters |
| `agnes-bridge` | Rust (cdylib) | napi-rs native addon exposing the core to JS |
| `agnes-library` | TypeScript | Typed schema DSL, query builder, `AgnesClient` |
| `agnes-cli` | TypeScript (Bun) | `push` / `pull` / `migrate` / `generate` |
| `example` | TypeScript | Runnable sample project |

## Why choose Agnes

- **You want caching that's actually safe.** Tag-based, SQL-derived invalidation
  means no stale reads and no manual key bookkeeping.
- **You want native performance** without leaving TypeScript.
- **You want the full schema workflow** — push, pull, migrations, and client
  generation — in one tool.
- **You want end-to-end types**, including nested relations, from schema to result.
